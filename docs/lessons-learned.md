# Lessons Learned — unleashed

| Date | Lesson | Rule/Action |
|:-----|:-------|:------------|
| 2026-03-16 | **Promoted untested c-28 to prod immediately after writing it.** User had to catch and correct this — stable version was c-27, c-28 had never been run. | **New code goes to alpha tier first.** alpha → beta (after testing) → prod (after burn-in). Never skip tiers, no matter how confident the code looks. |
| 2026-03-16 | **Said "admin terminal" without specifying which shell or how to open it.** User's system has bash, PowerShell 5, PowerShell 7, and cmd. "Terminal" is meaningless — user was rightfully frustrated. Gave wrong instructions multiple times, wasting the user's time while the system was in crisis. | **Always specify: exact shell name + exact click path to open it.** E.g., "Right-click Start → Terminal (Admin) — this opens bash. In that bash window, run..." Never assume which shell opens. |
| 2026-03-16 | **Used `/y` flag with `net stop` — not a valid flag.** Then gave `&&` (bash syntax) for PowerShell, then backtracked to `net stop` alone in bash. Three wrong commands in a row while user's machine was hung. | **Test Windows admin commands mentally before prescribing.** `net stop` has no `/y` flag — use `echo Y | net stop <service>` or just let the user type Y. Verify shell syntax matches the shell the user is actually in. |
| 2026-03-17 | **Heredoc commands silently fail in Git Bash → PowerShell.** User ran `powershell -Command - <<'PSEOF'` for services and scheduled tasks. Commands displayed in terminal but PowerShell never executed them. `-ErrorAction SilentlyContinue` hid errors. | **Never use `powershell -Command - <<'PSEOF'` heredocs.** Always use single-line `powershell -Command "..."` or `powershell -Command '...'`. Audit every powershell command in runbooks before committing. |
| 2026-03-17 | **Missed Teams startup registry entry while claiming "all clean."** Checked services, processes, packages (all gone) but didn't check HKCU\Run. User correctly called this contradictory with my startup analysis. | **When checking "is X fully removed?", check ALL vectors:** installed app, services, processes, scheduled tasks, AND startup registry entries. Removal checklist, not spot checks. |
| 2026-03-17 | **Told user their admin terminal lacked permissions when it clearly said "Administrator."** Real issue was heredocs failing silently, not permissions. User showed screenshot proving admin. Wasted trust. | **Never blame the user's environment without evidence.** If a command fails, verify the failure mode before attributing it to permissions. Read error messages carefully — "Access denied" from a non-admin session vs silent failure from a heredoc are completely different problems. |
| 2026-03-17 | **Pasted commands in chat expecting user to copy them.** User has said 10+ times they cannot copy from the chat window. I gave the correct commands but in the wrong medium — the user had to yell before I pointed to the runbook file and line numbers. | **Never paste commands in chat for the user to run.** Always put commands in a file first (runbook, script), then tell the user the file path and exact line numbers. If the file already has the commands, just point to it. |
| 2026-03-17 | **Running `new_repo_setup.py` without knowing PAT permissions.** Script ran all 12 local steps successfully then failed at step 13 (`gh repo create`). Wasted time resolving merge conflicts caused by the user creating the repo manually with GitHub defaults (README, LICENSE, .gitignore) that conflicted with the scaffold's versions. | **Before running a scaffolding script that touches external services, verify the required permissions first.** For `new_repo_setup.py`: either use `--no-github` and create the repo manually with NO template files (no README, no LICENSE, no .gitignore), OR verify the PAT has `createRepository` + `workflow` scope. Updated runbook 0901 v2.1 with this guidance. |
| 2026-03-18 | **Applied proxy fix to all three t-files instead of promoting the alpha.** User had to correct: promotion means the alias moves to the verified file, not backporting changes to old versions. Defeats the entire tier system. | **Promotion = move the alias forward.** Update bash_profile to point at the new file, mark old files as historical. Never edit old version files to add new features. |
| 2026-03-18 | **`GIT_SSL_BACKEND` is not a real git environment variable.** Set it in the env, it showed up in `env` output, but git silently ignored it and kept using schannel. Wasted a full Codex verification cycle. | **Use `GIT_CONFIG_COUNT`/`GIT_CONFIG_KEY_0`/`GIT_CONFIG_VALUE_0` for git config overrides via environment (Git 2.31+).** The `-c` flag works on the command line but there's no `GIT_SSL_BACKEND` env var. |
| 2026-03-18 | **Codex CLI on Windows uses PowerShell, not bash.** Wrote verification prompts with bash syntax (`env | grep`). Codex translated to PowerShell equivalents. The parent shell (Git Bash) is irrelevant — Codex picks its own sandbox shell. | **Codex prompts must use PowerShell syntax.** The execution chain is: Git Bash -> python wrapper -> cmd /c codex.cmd -> PowerShell. Bash is gone by the time Codex executes. |
